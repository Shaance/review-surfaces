# Agent goal: release quick-wins uplift — the first command tells the truth, one PR per phase

Goal: before the owner runs the first `npm publish`, close the small set of high-leverage gaps that five uplifts of development surfaced but never scheduled — every one is a real observed incident, not speculation, and every one is deliberately small. Two themes: **the first command a stranger types must tell the truth** (range resolution, working-tree honesty, portable artifacts), and **the package's first touch must be frictionless** (`--version`, Node guard, npm-page rendering, publish safety, repo hygiene). Each phase is its own PR, simplified, reviewed, and landed on `main` before the next begins (same discipline as `docs/history/POLISH_UPLIFT_GOAL.md`).

This builds on the five completed uplifts (PRs #47–#77). Nothing here may regress them: cold-start correctness, spec-less mode, the legibility budget, map/tour agreement, byte-determinism, and the committed example packet are the foundation.

Use these as the contract:

- This file's **Quick-wins evidence log** below — every item cites the incident or verified gap it closes.
- `features/review-surfaces.feature.yaml` — the requirements ledger. **Phase 0 creates the spec entries.** Preserve Acai IDs in tests, notes, and artifacts.
- `AGENTS.md` and `.agents/skills/composed-review-loop/SKILL.md` — working rules and the review/landing workflow.

## Standing constraints

- Everything runs locally with `--provider mock` (network: git only). Merge gate: clean review verdict plus `pnpm run local-gate` green.
- Byte-deterministic artifacts, redaction before every render, model-and-strict-schema-grow-first (required-field growth forces stale-artifact rebuild — the NARRATIVE.1 precedent).
- **New standing rule for this uplift — the quick-win budget:** every item below is sized at half a day or less. If implementation reveals an item is bigger, ship the honest subset that fits the budget and record the remainder as a note in the PR — never grow the scope. An uplift named "quick wins" that takes three weeks has failed regardless of code quality.
- `npm publish` remains the owner's single manual step — prepared, verified, never run by an agent.

## Quick-wins evidence log (2026-06-12)

Method: verified against `main` (`69a5001`, v0.2.0) by source inspection and live CLI runs, plus the incident history of the five uplifts (PRs #47–#77).

1. **A missing base ref silently reviews the wrong thing.** `--base` defaults to a hard-coded `origin/main` (`src/cli/index.ts:293,672`, help text `:3093`, `action.yml:32`, init/packet command templates). When it does not resolve — a `master`-default repo, a repo with no remote, a CI shallow clone — `collectChangedFiles` falls back to a **bare working-tree diff** (`src/collector/git.ts:74-81`) and the only trace is a stderr diagnostic that is by design "never persisted" (`git.ts:21-23,51-61`). A stranger on a `master` repo gets a confidently wrong (usually empty) verdict from the README quickstart. Our own CI needed `fetch-depth: 0 # need origin/main` comments (`.github/workflows/ci.yml:13,97`) — proof adopters will hit this in Actions too.
2. **Working-tree files contaminate pinned-range reviews.** `collectChangedFiles` unions `git status --untracked-files=all` into **every** run, even an explicit historical range (`src/collector/git.ts:88-104`). Observed 2026-06-12: the pinned `aefc63c~8..aefc63c` dogfood counted 104 files instead of the range's 99 because five uncommitted working-tree files leaked in; the README screenshots had to be regenerated from a clean clone to work around it. The model already tags these files (`source: "working_tree"`, `git.ts:14-18`) but no rendered surface ever says "this review includes uncommitted changes".
3. **Artifacts embed machine-specific `../` path chains when `--out` is outside the repo.** `pointersForPacketPath` writes `path.relative(cwd, packetPath)` into rendered output (`src/render/comment.ts:198-203`). Observed twice on 2026-06-12: the example-packet artifacts leaked relative `/tmp` chains and a cockpit screenshot header showed `../../tmp/rs-clone/...`; both were worked around by re-running from a clean clone with an in-repo `--out` instead of fixing the renderer. An out-of-repo artifact dir is a legitimate choice and currently produces non-portable artifacts.
4. **`review-surfaces --version` is an error.** Verified live: `Unknown command: --version`. It is many people's literal first command after install (`npx review-surfaces --version`). Bare no-args output is good (help leads with `review-surfaces 0.2.0`), but the canonical flag fails.
5. **Nothing guards the Node floor at runtime.** `engines: ">=22"` is declared but npm/npx does not enforce it, and `dist` targets ES2022 — on Node 18/20 the CLI parses fine and then dies mid-run with a bare `TypeError` on some newer API instead of one actionable line. `bin/review-surfaces.js` is plain CJS and already the pre-dist entrypoint — the natural place for a five-line guard.
6. **The README is not npm-page-proof and the package metadata is thin.** All three screenshots and the example-packet link use repo-relative paths (`docs/images/...`, `docs/example/README.md` — `README.md:22,58,74,82`); relative images render unreliably on npmjs.com and on mirrors (socket.dev, deps.dev). `package.json` has `repository` and `keywords` but no `homepage` and no `bugs`, so the npm sidebar loses its two highest-click links.
7. **`prepublishOnly` only builds — the one manual step can ship a red gate.** `package.json:37` runs `pnpm run build`, nothing else. The failure class is proven: during Phase 3 a red gate was pushed as `64e6d95` because the gate's exit code was masked (piped through `tail -3`) and the gate had to be caught after the fact. The owner's single `npm publish` should be physically incapable of repeating that.
8. **The quickstart writes an un-ignored directory into the user's repo and never says so.** `review-surfaces all` creates `.review-surfaces/` (a ~100 KB cockpit HTML among other artifacts) in a stranger's working tree with no hint that it is local-first output worth gitignoring; the next `git add -A` commits it. This repo ignores it with tracked exceptions (`.gitignore:23-27`), so it can never observe the gap itself.

Observed but deliberately out of scope (the quick-win budget excludes them): pipeline runtime, Windows CI, mermaid dark-mode theming, SVG pan/zoom, publish automation/provenance workflows, auto-editing a user's `.gitignore`, and any redesign of working-tree review semantics beyond removing the silent paths.

## Phases (strict order, one PR each)

1. **Range truth — the first command reviews exactly what it claims.**
   - **Base auto-resolution (closes 1).** The default base becomes an explicit resolution chain — `origin/HEAD` → `origin/main` → `origin/master` → `main` → `master`, first ref that resolves, merge-base semantics unchanged. An **explicitly passed** `--base` that does not resolve is a hard error (non-zero exit, no artifacts written): name the ref, say shallow clones are the usual cause, and give both fixes (`fetch-depth: 0` for Actions, `git fetch --unshallow origin <ref>` locally). An exhausted auto-chain errors the same way, listing what was tried. The silent working-tree fallback dies; whether an explicit opt-in worktree mode replaces it is an implementation decision, but no *silent* path survives. The resolved range (ref + short SHA for both sides) is printed in the `all` summary and persisted as model state in the manifest/human-review header — the current "never persisted" stderr diagnostic (`git.ts:21-23`) is exactly the bug. README quickstart simplifies to bare `npx review-surfaces all`; help text, init templates, and the packet follow-up command line update to match; `action.yml` keeps its explicit default (its failure mode becomes the actionable error) and the action docs gain one `fetch-depth` line.
   - **Working-tree honesty (closes 2).** The `git status` merge happens only when the requested head resolves to the checked-out `HEAD`; a pinned head gets a pure range diff — with an ACID test reproducing the 104-vs-99 incident (dirty tree + pinned range ⇒ byte-identical artifacts to a clean-tree run). When the merge does contribute files, every human surface carries one deterministic line — "includes N uncommitted file(s) (working tree)" — in the `all` summary, the `human_review.md` header, the cockpit header, and the sticky when nonzero. The model already knows (`source: "working_tree"`); this is surfacing, not new analysis.
   - **Portable artifacts (closes 3).** No rendered artifact may contain a `../` chain or an absolute path, regardless of where `--out` points. Pointers compute repo-root-relative with a stable fallback when the artifact dir lies outside the repo (exact form is an implementation decision; the test is the contract). ACID test: run with `--out` in a temp dir outside the repo, scan every rendered artifact for `../` and absolute-path patterns, and assert the artifact bytes match an in-repo-out run.
   - Hard rules: ACID-named tests first; schema growth is additive and strict-schema-first. Dogfood matrix beyond `pnpm run local-review`: a got clone with its default branch renamed to `master`, a `--depth 1` shallow clone, a no-remote local repo, a dirty-tree pinned-range run on this repo, and the standing pinned legibility ranges (`aefc63c~1..aefc63c`, `aefc63c~8..aefc63c`; got `a5b76bffb` `HEAD~3` and `HEAD~40`) to prove no regression.

2. **Package first touch — the first five minutes after `npm install`.**
   - **`--version` (closes 4).** `review-surfaces --version` (and the bare `version` command) prints exactly the help header line (`review-surfaces 0.2.0` from the shared `VERSION` constant) and exits 0. The existing version-sync tests extend to cover it.
   - **Node floor guard (closes 5).** `bin/review-surfaces.js` checks `process.versions.node` before spawning dist: below 22 ⇒ one actionable stderr line ("review-surfaces requires Node >= 22; you are running vX — engines is declared but npx does not enforce it") and exit 1. The shim must stay parseable by old Node (plain CJS, no modern syntax) and the no-dist `run` fallback keeps working. Assert via source-assertion test (the `distribution.test.ts` pointer-ordering tests are the precedent).
   - **npm-page-proof README + metadata (closes 6).** Image and example-packet links become absolute (`raw.githubusercontent.com/.../main/docs/images/...` for images, GitHub blob URLs for docs); `package.json` gains `homepage` and `bugs`. Test: README contains no relative `](docs/` or `](./` links, metadata fields present. Decide at implementation whether `docs/images` then leaves the `files` allowlist (smaller tarball) — the pack smoke test already measures the consequence.
   - **Publish runs the gate (closes 7).** `prepublishOnly` becomes `pnpm run local-gate`, so the owner's one manual step cannot ship a red gate — this encodes the `64e6d95` lesson into the release mechanics. Test asserts the script wiring; CHANGELOG's 0.2.0 section gets one line saying publish runs the full gate.
   - **Ignore hint (closes 8).** After writing artifacts, if the out dir is inside the repo and is neither git-ignored (`git check-ignore`) nor already tracked (`git ls-files` non-empty under it), print exactly one stderr hint: add it to `.gitignore`, artifacts are local-first. Stderr only — the stdout ordering contracts (HUMAN_REVIEW.15, pointer-last) are asserted by tests and must not move. This repo (ignored + tracked exceptions) stays hint-free, asserted.
   - Hard rule: this whole phase is small by design — if any single item exceeds its budget, ship the rest and log it; do not let the phase swell.

## Per-phase loop

Identical to the polish uplift: branch `quickwin-phase-<n>-<slug>` from fresh `origin/main`; implement with ACID-named tests first; `pnpm run local-gate` green **before** pushing; dogfood via `pnpm run local-review` plus the phase's dogfood matrix; remove the phase's ACIDs from `allow_missing`; simplify pass; PR with phase number, ACIDs, validation commands, dogfood verdict; review and land per repository gates; next phase only after merge.

Process gotchas this cycle proved the hard way — read before starting:

- **Never pipe `pnpm run local-gate` through `tail`/`head`** — it masks the exit code; a red gate shipped as `64e6d95` exactly this way. Run it bare, check the exit, then push as a separate step.
- **Reviews race pushes.** Before acting on review findings, check the review's `commit_id` against the current head; findings against a prior head are resolved as stale and the review re-triggered for the new head.
- **`git checkout -- .review-surfaces` before staging** — never `git add -A` over regenerated artifacts.
- **Committed artifacts and screenshots regenerate only from a clean clone with a frozen `--now`** — working-tree contamination (evidence item 2) is invisible until you count files.

## Success condition

All phases merged; gate back to `max_missing: 0, allow_missing: []`; post-merge empty-diff `local-gate` green on `main`. Concretely: bare `npx review-surfaces all` on a `master`-default clone reviews the same range as an explicit-base run and prints + persists the resolved range; a bad explicit base exits non-zero with the shallow-clone fix and writes nothing; a dirty-tree pinned-range run is byte-identical to a clean-tree run, and a dirty `--head HEAD` run says "includes N uncommitted file(s)" on every human surface; `--out` outside the repo yields artifacts with no `../` or absolute paths; `--version` works; old Node gets one clear line instead of a mid-run TypeError; the README renders correctly on the npm page with `homepage`/`bugs` in the sidebar; `prepublishOnly` runs the full gate; and a stranger's first run hints once about gitignoring the artifact dir — with `npm publish` still the owner's single manual step.
