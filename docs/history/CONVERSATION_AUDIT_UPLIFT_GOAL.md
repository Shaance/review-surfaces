# conversation-audit uplift — goal contract

**Status: proposed (2026-06-16).**

Scope: make the methodology surface genuinely *read the raw agent conversation* (Claude Code, Codex, OR Cursor) and use an LLM provider leaf — validated against deterministic evidence — to answer the user's item 4 (judge the methodology/whole interaction) and item 5 (compute conversation-derived test/risk gaps), while keeping the "deterministic shell, probabilistic leaves" discipline intact.

This file is the complete contract. It is implemented by an agent with **no prior conversation context** — every fact, path, and decision needed is in this document. Use `features/review-surfaces.feature.yaml` (the Acai requirement ledger), `docs/review-surfaces-trd.md`, `schemas/review_packet.schema.json`, and `AGENTS.md` as the standing sources of truth. Preserve Acai IDs such as `review-surfaces.METHODOLOGY.6` in code comments, tests, and artifacts.

## Verdict: has this been tackled?

**PARTIALLY.**

A deterministic keyword-grep methodology scaffold exists and satisfies the *literal* feature-spec requirements `METHODOLOGY.1-5` and `RISK.1-5` at a **shallow** level. The user's *deeper intent* — genuinely reading the raw agent transcript to judge whether the workflow made sense (item 4) and to compute conversation-derived test/risk gaps (item 5) — is **not built**.

**What EXISTS and is solid (keep it):**
- A real, evidence-bound trust gate: validation-success/failure claims in the conversation ("tests passed", "lint clean") are matched against the actual command transcripts and split into `verified_claims` vs `claims_without_evidence` by an *exact normalized command match* (`src/methodology/methodology.ts:62-68, 172-230`). The negation guards exclude hedges/TODOs. This is the one genuinely evidence-bound part and is the only conversation signal that reaches risks (`src/risks/risks.ts:150-163`).
- Secret redaction of every event summary at ingest via `redactConversationSummary` → `redactSecrets` (`src/methodology/methodology.ts:122,134,147,264-266`), and a locally-written normalized log `inputs/conversation.normalized.jsonl` (`src/methodology/methodology.ts:151-156`).
- Missing-log non-fatality: a missing `--conversation` returns a finding with `quality_flags:["conversation_log_missing"]`, never throws (`src/methodology/methodology.ts:42-56`).

**What is SHALLOW:**
- `considered` / `research` / `decisions` / `unchallenged_assumptions` / `skipped_checks` are **pure keyword grep** via `pick(events, [...])` (`src/methodology/methodology.ts:73-79`). `pick()` is first-match-wins in raw event order, hard-capped at 12, echoes back the matched summary verbatim, no ranking, no scoring, no reasoning (`src/methodology/methodology.ts:158-170`). On a long session the 13th-onward signal is silently dropped.
- `unchallenged_assumptions` greps the literal tokens `"assume"/"assumption"` (`:76`) — it surfaces assumptions the agent **explicitly named**, the exact inverse of item 4(c) "what might NOT have been challenged". Absence is structurally undetectable by grep.
- `skipped_checks` greps `"skip"/"not run"/"could not"` (`:77-78`) — it detects only checks the transcript *names* as skipped; a check that should have run but was never mentioned produces nothing.
- The summary is a bare event count: `Methodology extracted N event(s) from <path>` (`:71`). No workflow-soundness verdict, no importance ranking anywhere.

**What is ABSENT:**
- **No raw-transcript adapter.** `parseConversationFile` accepts exactly three *pre-normalized* shapes, dispatched by file extension only: `.jsonl` already shaped `{id,actor,kind,summary}`; `.yaml/.yml` with `{events:[...]}`; or plain `user:/assistant:` text (`src/methodology/methodology.ts:110-149`). No code reads a Claude Code session `.jsonl`, a Codex rollout, or a Cursor export. `grep -r conversation-format src/` returns zero hits.
- **No tool-call evidence.** `ConversationEvent` is `{id,actor,kind,summary}` (`src/methodology/methodology.ts:24-29`) — no field for tool name, command, file path, or exit signal, and no parser reads `tool_use`/`tool_result`/`function_call` blocks. So item 4(b) "what research was done" and item 5(b) "HOW it was tested" cannot be grounded in the agent's actual actions, only its prose.
- **No LLM leaf reads the conversation.** `buildMethodology` has zero provider wiring. The two LLM stages that *touch* methodology (`runNarrativeStage` at `src/llm/reasoning.ts:611-685`; enrichment at `src/llm/provider.ts:604-612`) prompt on **summary strings only** and never receive the `ConversationEvent` stream. So no provider ever reads the raw log — contradicting decision D1.
- **No conversation-derived "what/how tested + which steps untested".** Of `analyzeRisks`'s five risk items, four are diff/requirement-derived from `evaluation.results`; the only conversation-derived one is `methodology.claims_without_evidence` (`src/risks/risks.ts:84-163`). `test_gaps`/`missing_automatic_tests`/`missing_manual_checks` are all derived from weak requirement results, capped at 20, and every weak requirement lands in *both* missing-lists with canned text (`src/risks/risks.ts:193-221`). No "HOW tested" axis exists; `analyzeRisks` never receives a `StructuredDiff` or `SemanticChangeFacts` (`src/risks/risks.ts:71-76`).
- **No surface for a deep audit.** Packet section 5 renders only `summary` + `verified_claims` + `claims_without_evidence` + `skipped_checks` (`src/render/packet.ts:245-255`); `considered`/`research`/`decisions`/`unchallenged_assumptions` are dead model fields. The human cockpit's three-questions strip reads overreach/weakening/unbacked only (`src/human/render-html.ts:358-387`).

## Theme

Turn the methodology surface from **keyword-grep over a hand-authored normalized log** into a **real audit of the raw agent transcript** that answers the user's two questions:

- **Item 4 — methodology / whole interaction:** from the raw conversation, judge (a) what was considered, (b) what research was done, (c) what might NOT have been challenged — and decide whether the workflow made sense and whether something important was skipped.
- **Item 5 — risk / tests:** from the raw conversation, compute (a) what was tested, (b) HOW it was tested, (c) which workflow steps lack manual/automatic tests.

The LLM (provider leaf) is the **primary analysis engine** reading the redacted, normalized conversation; the deterministic layer keeps three jobs so "deterministic shell, probabilistic leaves" (TRD 3.3) still holds: (1) the claim-vs-command **trust gate** stays deterministic and marks claims verified-vs-unbacked; (2) a deterministic **keyword fallback** runs when no provider is available (offline/CI/mock) so the pipeline never breaks and golden tests stay byte-stable; (3) **validation/anchoring** checks every LLM finding against real evidence (event ids, file paths, the diff) before it is trusted, exactly as intent/narrative already validate LLM anchors. Net: the LLM reads and reasons; determinism gates trust and guarantees offline reproducibility.

**Operating reality (read before sizing the build).** D1 makes the LLM leaf the primary *analytical* path **when a provider is configured** (`agent-file`/`ai-sdk`). But the repo default provider is `mock` (`src/cli/index.ts:751` — "The default mock provider returns not-ok"), and every dogfood / CI / golden run is mock. So the **degraded keyword fallback + `methodology_analysis_degraded` flag (D2) is the de-facto default OUTPUT** in shipped runs, while the rich leaf only runs under `agent-file` (with a real `--agent-input`) or `ai-sdk`. Two consequences the implementer MUST hold: (1) every deep-audit surface (packet section 5/6/7, cockpit, reviewer questions) must render gracefully in BOTH states — full audit when a provider returns structured output, the degraded fallback + loud flag otherwise — so the cockpit's *normal* output is never mistaken for the deep audit; (2) the rich leaf path is exercised ONLY by the test FAKE provider (the `stubProvider` pattern, `tests/human-narrative.test.ts:48-53`) or a real `--agent-input` file. **Bare `--provider agent-file` with NO `--agent-input` is NOT sufficient:** `agentFileProvider` returns `{ok:false,reason:'missing_agent_input'}` when `--agent-input` is absent and `{ok:false}` on a parse error (`src/llm/provider.ts:87-110`), which correctly DEGRADES (D2). Do not assume `--provider agent-file` alone gives the deep audit in dogfood. Build effort stays proportional: the deterministic shell is the common path, the leaf is the configured-provider path.

## Decisions (settled)

These were settled with the user via Q&A on 2026-06-16. They are **closed** — implement them, do not reopen or offer them as options.

- **D1 — LLM-required-by-default for conversation analysis.** The provider leaf is the PRIMARY engine for items 4 & 5; the deterministic keyword extraction is an explicit FALLBACK only. (See "Operating reality" above for how this coexists with the mock default.)
- **D2 — No-provider behavior: degrade, never fatal.** With no LLM provider (mock/offline/CI/tests), run the deterministic keyword fallback and set a LOUD quality flag (`methodology_analysis_degraded` / "no LLM provider; deep audit not run"). Tests/dogfood use a FAKE provider returning canned, deterministic structured output (mirroring the mock used for intent/narrative) so golden tests stay byte-stable offline.
- **D3 — Long conversations: map-reduce.** Chunk the salience-ordered transcript, LLM-summarize each chunk, merge with a DETERMINISTIC merge order; bound the chunk count / LLM calls (reuse `src/llm/reasoning.ts` `chunk()`/`MAX_*_PER_BATCH`). If still over the hard cap, SIGNAL truncation explicitly (quality flag + visible "audit was partial"), never silent. Content-hash CACHE so an unchanged conversation is not re-summarized.
- **D4 — Conversation discovery: auto-discover with explicit override.** Scan the harness session stores (`~/.claude/projects`, Codex sessions, Cursor storage), pick the session matching THIS repo/branch; `--conversation` ALWAYS wins. Discovery is READ-ONLY, redacts before any LLM call, never copies raw into artifacts (PRIVACY.1), and ANNOUNCES which session file it picked so the user can correct a wrong match. (Concrete matching contract specified in Phase 5b.)
- **D5 — Verdict authority: advisory unless validated.** LLM findings surface as reviewer questions/notes; they become blockers ONLY when an independent deterministic check (missing transcript, unmapped change, semantic-diff risk) also fires. The deterministic verdict stays reproducible. LLM-cited anchors (event ids, paths) are VALIDATED and dropped/demoted if invalid, exactly like intent/narrative anchor validation.
- **D6 — Audit depth: cross-reference conversation with the diff.** Tie conversation turns to changed files and flag mismatches. v1 MUST flag ALL FOUR signals: (a) risky code changed (auth/crypto/secrets/input-validation) with no security discussion; (b) implementation changed with no test added/run that exercises it; (c) exported API / JSON schema / public contract changed with no compatibility/migration discussion (reuse `src/risks/semantic-diff.ts` `test_weakening`/API-surface/schema-contract facts); (d) dependency/lockfile/CI/config changed with no rationale (reuse `src/risks/dependency-facts.ts` `computeDependencyFacts` + `src/risks/config-facts.ts` `computeConfigFacts` — see Phase 3).
- **D7 — Conversation scope: a single working SESSION** that produced base..head. Do NOT stitch multiple sessions. Discovery matches the one session for this branch (selection + tie-break in Phase 5b).
- **D8 — Transcript scope: full transcript INCLUDING tool calls.** Adapters extract `tool_use`/`tool_result` (commands run, files read/edited), not just user/assistant text. Tool calls are the primary evidence for item 5.
- **D9 — Normalized-log storage: gitignore by default.** `inputs/conversation.normalized.jsonl` stays local; opt-in to include. Keeps conversation content out of git while preserving local provenance + content-hash caching. (Gitignore coverage is qualified — see "Current state" and the `.review-surfacesignore` entry added in Phase 1. Gitignore is *necessary but not sufficient* for privacy — see PRIVACY.7(a)/Phase 1 for why the persisted normalized log must itself be redact-before-bound, not merely gitignored.)
- **D10 — Multi-harness: Claude Code, Codex, AND Cursor are ALL first-class** — pluggable adapter registry, auto-detect + `--conversation-format` override, each with a fixture + test, none deferred.

## Target design

Built as **incremental, independently-shippable PHASES**, each one a single coherent capability that can land — and be reverted — on its own. The discipline of "deterministic shell, probabilistic leaves" is preserved: deterministic extraction + evidence binding stay authoritative; the leaf only adds bounded judgement, and its output is not proof until evidence validation accepts it.

**Phase dependency map (so each increment is genuinely shippable AND truncation-honest on its own):**
- Phase 0 (spec) → prerequisite for all.
- Phase 1 + Phase 1.5 (adapters + the event seam, same PR or immediate successor) → prerequisite for Phase 2.
- Phase 2 (methodology leaf) carries its OWN minimal hard cap + `conversation_truncated` flag (so it is truncation-honest the moment it ships — see the Phase 2 "Bounded event stream" bullet), and depends only on Phase 1.5.
- Phase 3 splits into a deterministic half and an LLM half: **Phase 3a** (the deterministic D6 facts → risks + the four signals as deterministic checks) depends ONLY on Phase 1.5 and the existing fact modules — it has **no Phase-2 dependency** and is the highest-confidence independently-shippable slice, so it may land right after Phase 1.5 (in parallel with Phase 2). **Phase 3b** (the LLM `CONV-GAP-*` producer + `tested_how`) depends on Phase 2's validated `cross_ref_flags`. Land 3a first; 3b after Phase 2.
- Phase 4 (surfacing) depends on Phase 2/3 model fields.
- **Phase 5a (map-reduce + salience + content-hash cache, D3) and Phase 5b (auto-discovery + single-session selection + slug transform, D4/D7) are TWO INDEPENDENT phases that share NO code dependency** and may ship in EITHER ORDER after Phase 2 (full independence argument in "## (Phase 5a + 5b interaction)" below). 5a REPLACES Phase 2's naive head-cap with salience-ordered chunking; 5b folds the discovered-session identity into the cache signature. Their ACIDs stage separately in `allow_missing` so each removes independently.

- **kind mapping:** `text` block → `message`; `tool_use` block → `tool_call` (capture `name`→`tool`, input→`command`/`file`); `tool_result` block → `tool_result`. Unknown block type → `message` summary, never throw.
- **text extraction:** join text blocks for the summary; for `tool_use` summarize as `<name>(<bounded input>)`; for `tool_result` summarize the bounded output.
- **deterministic id:** from the envelope `uuid` (fallback: `<session-id>-<line index>`); for multi-block lines, suffix the block index.
- **redaction:** every extracted field (text, tool_use input, tool_result output) through `redactSecrets`; bounded excerpts through `redactForArtifact` (redact-before-bound).

### Codex CLI adapter (`src/conversation/adapters/codex.ts`, new)

- **On-disk shape:** the rollout/session transcript (JSONL/JSON of response items) under the Codex CLI session dir (e.g. `~/.codex/sessions` or the repo's logs) — items of type `input_text`/`output_text`/`function_call`/`function_call_output`.
- **detect():** items carry response-item types like `input_text`/`output_text`/`function_call`/`function_call_output` (and/or item/call ids).
- **actor mapping:** `input_text → user`; `output_text → assistant`; `function_call → assistant` (kind `tool_call`); `function_call_output → tool` (kind `tool_result`).
- **kind mapping:** `input_text`/`output_text` → `message`; `function_call` → `tool_call` (capture name + arguments as `command`/`file`); `function_call_output` → `tool_result`.
- **text extraction:** item text for messages; `<name>(<bounded arguments>)` for `function_call`; bounded output for `function_call_output`.
- **deterministic id:** from the item `id` or `call_id` + index.
- **redaction:** as above; tolerate schema drift across CLI versions (best-effort field reads, never fatal on unknown fields). A secret in a `function_call_output` is treated exactly like a Claude Code `tool_result` secret (feeds the PRIVACY.7 fold).

### Cursor adapter (`src/conversation/adapters/cursor.ts`, new)

- **On-disk shape:** the exported chat/composer JSON — a conversation object with a `messages` array of `{role, text, ...}` plus tool/code-edit entries (exportable from the Cursor UI or its workspace storage). Tolerate both UI-export and workspace-storage variants.
- **detect():** a JSON object with a `messages` array of `{role,text}` items (and/or composer/bubble ids).
- **actor mapping:** `role → actor`; tool/edit entries → `tool`.
- **kind mapping:** `text` → `message`; tool/code-edit entries → `tool_call`/`tool_result` capturing the touched `file`.
- **text extraction:** message `text` for the summary; for edits, summarize the touched file + bounded edit body. **The code-edit body is a high-exposure channel** (pasted keys, `.env` diffs) and is the Cursor-specific evidence surface — it MUST route through `redactSecrets` at normalization and through `redactForArtifact` before bounding (redact-before-bound), exactly like the Claude Code `tool_result` body, and a blocked-kind secret there feeds the PRIVACY.7 fold.
- **deterministic id:** from the message id / bubble id, fallback index.
- **redaction:** as above, with the code-edit body explicitly covered (see test plan PRIVACY.7).

## Privacy & determinism discipline

Hard constraints — non-negotiable:

- **PRIVACY.1 / TRD 14.1:** raw conversation logs are never copied into public artifacts by default. Store normalized summaries + event ids; hash sensitive excerpts; require explicit config to include raw excerpts. Raw copies live only under `.review-surfaces/inputs/conversation.raw.*` (in `.review-surfacesignore`); normalized output under the gitignored output dir, with `.review-surfaces/inputs/conversation.normalized.*` added to `.review-surfacesignore` (Phase 1) so it is excluded independent of `--output-dir`. **Gitignore is necessary but NOT sufficient:** the persisted normalized log is itself a persisted surface, so each written field routes through `redactForArtifact` (redact-before-BOUND) and any field where `containsBlockedRedaction` fires is stored hash-only by default (raw masked excerpt only on explicit opt-in) — a blocked secret must not land verbatim, or retain surrounding context, even in a local gitignored artifact (PRIVACY.7(a), Phase 1). For an auto-discovered (non-repo) session, the persisted `conversation` `EvidenceRef` carries the harness label + content hash + repo-relative normalized-log path ONLY — never the absolute home-dir session path (it would fail `isSafeRepositoryPath` and leak the path); the absolute picked path is announced on stderr only (Phase 5b).
- **PRIVACY.2 / PRIVACY.7 (b):** secret-scan/redact BEFORE any remote LLM call. The leaf routes through `ReasoningProvider.generateStructured`, inheriting privacy-block-before-remote and the high-severity `redactSecrets(prompt).blocked` short-circuit (`src/llm/provider.ts:128-148`) — but that short-circuit is NOT the sole gate. The conversation block signal is OR'd into `privacy.remote_provider_blocked` IN `collect.ts`, BEFORE the privacy object is assembled at `src/collector/collect.ts:361-367` (NOT later in `stages.ts`, where the provider has already read the frozen value via `reasoningProviderFor`/`reasoningOptionsFor` at `stages.ts:65,73`), in **Phase 2** — the same phase that first sends conversation events to `ai-sdk` — so the gate (`src/core/gate.ts:68`) blocks an `ai-sdk` run carrying a `tool_result` secret. The methodology leaf MUST NOT reach `ai-sdk` before this fold lands. Only `providerMakesRemoteCall(=ai-sdk)` is blockable; mock/agent-file stay offline.
- **PRIVACY.6:** every persisted/postable surface redacts and exposes a high-risk block signal. Beyond the gate fold, the conversation secret scan is ALSO merged into `privacy.secret_findings` (currently diff-only, `src/collector/collect.ts:367`) so a persisted `human_review.json`/packet that held a blocked-kind `tool_result` secret reports blocked material — not a clean `secret_findings` next to a `remote_provider_blocked=true` (PRIVACY.7(b)). Route every chunk/excerpt/window AND the persisted normalized log through `redactForArtifact` (redact-before-bound) and feed `containsBlockedRedaction` so a persisted surface can detect (and mark) that it held blocked material.
- **Determinism:** no `Date.now()` / `new Date()` / `Math.random` in any new conversation code — timestamps come from event data or the injectable `NowProvider` (`src/collector/collect.ts:39-48`); the discovery tie-break uses the latest IN-SESSION event timestamp, NOT file `mtime`/wall-clock; map-reduce merges are deterministically ordered with the explicit key (first-cited event `raw_index`, then normalized text — see Phase 5a); event ids are deterministic per adapter. **Cache-signature inputs across phases:** the conversation FILE is already content-hashed via `hashFlagInputs`; Phase 1.5 adds the resolved adapter label; Phase 5b adds the CONTENT HASH of the auto-discovered session file (NOT path-only — a path-only fold would re-use a stale audit when the live session grows). A live-growing in-session run legitimately busts the cache (correct — the session must be re-audited); a static fixture store recurs identically across reruns. Do not "fix" the live-session cache-bust by folding path-only.
- **Byte-deterministic output for identical inputs;** validates against the checked-in strict schemas (`additionalProperties:false`), which run in the gate (`src/schema/json-schema.ts:38`). The mock provider stays a guaranteed no-op (`{ok:false}`); golden tests use a FAKE provider returning canned structured output (the `stubProvider` pattern, `tests/human-narrative.test.ts:48-53`). Bare `--provider agent-file` with no `--agent-input` returns `{ok:false}` and correctly degrades — it does NOT exercise the rich path.
- **Missing logs never fatal** (`METHODOLOGY.4`), including a failed auto-discovery.
- **LLM / agent-file output is NOT proof** until deterministic evidence validation accepts it: anchors are validated against the real event-id set (the NEW `knownEventIds` branch in `validateEvidenceRef`, with `conversation`-kind refs REQUIRING a known `event_id` — path membership alone is insufficient, see Phase 2 — `event_id` is unvalidated today) and the changed-file allowlist; invalid anchors demote (never silently drop); findings stay advisory (`llm_proposed:true`) and become blockers only when an independent deterministic check also fires (D5). The deterministic verdict stays evidence-derived.
- **Schema growth is coordinated and additive, and split by KIND of addition:** *enum/union value-set additions* go in `src/schema/review-packet-contract.ts` (a `PACKET_*` const + derived type) + the matching schema enum, kept in sync by `tests/schema-contract.test.ts` — this is the ONLY thing the contract file holds (it has no object interfaces). The NEW enums in this uplift are `tested_how` (`PACKET_TESTED_HOW`) and any constrained `workflow_findings.signal_kind`/`severity`. *Object-field / shape additions* go in the JSON-schema `$defs` (e.g. a new `$defs/WorkflowFinding`; the `Methodology`/`Risks` object additions; `tested_how` on `TestGap`/`MissingAutomaticTest`/`MissingManualCheck`) + the TS model files (`MethodologyModel` in `src/methodology/methodology.ts`; the inline `RisksModel` literals in `src/risks/risks.ts:43-67`) + the two renderers (`src/render/packet.ts`, `src/human/render.ts`/`render-html.ts`) + a focused test — NOT in the contract file. **House tighter-than-schema pattern (REQUIRED):** any NEW array field added to `MethodologyModel` (notably `workflow_findings`) is a REQUIRED `[]`-defaulted TS array (optional in the JSON schema), so every code path including the degraded fallback MUST populate it (at least `[]`). Each such addition is a single-PR coordinated change keeping `MethodologyModel`/`RisksModel` and the schema's required/optional sets in sync. The three risks gap arrays are inline anonymous element types — `tested_how` is added to all three inline literals plus the matching schema `$defs` (or extract them into named interfaces first; pick one). Prefer reusing existing fields (`unchallenged_assumptions`, `skipped_checks`, `RiskItem.category='workflow'/'testing'`, `EvidenceRef.kind='conversation'/'command'`, `event_id`/`llm_proposed`/`validation_status`) before adding new ones.
