# Agent goal: implement the visual value uplift (M9–M11), one PR per phase

Goal: implement the selected ideas of `docs/visual-value-brainstorm-2026-06.md` — ideas **1–9 and 13** (the P3 carryovers 10–12 are explicitly out of scope) — sequentially, with each phase delivered as its own PR that is simplified, reviewed, and landed on `main` before the next phase begins.

This builds directly on the completed next-value uplift (`NEXT_VALUE_UPLIFT_GOAL.md`, PRs #53–#62). Everything that uplift shipped — the PR surface with sticky comment and since-last-review delta, queue ranking v2 with `ranking_reasons`, coverage-delta evidence, the time-budgeted mode, the seeded-regression eval harness, fear-class facts, the single-file HTML cockpit, team policy, and intent synthesis — is the foundation; nothing here may regress it.

Use these as the contract:

- `docs/visual-value-brainstorm-2026-06.md` — the brainstorm: per-idea rationale, value/effort scoring, **Approach** blocks (the contract for how each idea fits the architecture), the anti-ideas list, and milestone sequencing.
- `features/review-surfaces.feature.yaml` — the source-of-truth requirements ledger. **The spec entries for this work do not exist yet; Phase 0 creates them.** Preserve Acai IDs in tests, notes, and artifacts.
- `AGENTS.md` and `.agents/skills/composed-review-loop/SKILL.md` — working rules and the review/landing workflow for this repo.

## Standing constraint: everything runs locally, with zero GitHub Actions

GitHub Actions on this repo is currently billing-blocked, and target users of this tool may have no CI minutes at all. Therefore, for every phase of this goal:

- All artifacts a phase ships must be producible **entirely locally** with `--provider mock` (network use: git only). CI is a distribution channel for surfaces, never the only way to produce or verify them.
- The per-phase dogfood and merge gates run locally. Phase 1 ships the scripts that make this one command; until then, run the underlying commands directly.
- While Actions remain unavailable, the merge gate is: a clean review verdict from the PR reviewer **plus** the full local gate green (`lint`, `typecheck`, full `test`, `determinism-check`, and the strict empty-diff self-dogfood). This codifies the gate already used to land Phases 2–6 of the previous uplift.

## Two trust rules every visual must obey

These apply to every phase below, in addition to the standing rules (byte-deterministic artifacts, redaction before every render, model-and-strict-schema-grow-first, mock provider by default):

1. **Every node, edge, gutter mark, and chip cites a deterministic fact** in the packet (import graph, blast-radius, coverage, lens assignment). A visual is evidence with a layout, never an illustration.
2. **Caps are rendered, never silent.** Any bounded visual (node caps, halo caps, uncovered-line lists, trend history) shows an explicit overflow marker ("+ N more files", "history begins at round 3") — the same stance as the import graph's `truncated` flag.

## Phases (strict order, one PR each)

1. **Local review loop** — brainstorm idea 13, first on purpose: every later phase's dogfood step consumes it. `scripts/local-review.sh` (exposed as `pnpm run local-review`): build, run `all --provider mock` against `<base>..<head>` (default `origin/main..HEAD`), render the sticky-comment preview to a local file via `comment --format sticky`, render `human --format html`, run `validate --surface all`, and print an index of the artifacts to open. Accepts a previous-round input (`--previous <dir>` or auto-detected last run) so `since-last-review` — and later the Phase 4 rounds ledger — works from local prior packets, not only CI artifacts. `scripts/local-gate.sh` (exposed as `pnpm run local-gate`): `lint`, `typecheck`, full `test`, `determinism-check`, and the strict empty-diff self-dogfood (`all --base HEAD --head HEAD --strict` — the documented red-main footgun when a phase drops allowlist entries). Document both in `README.md` and `AGENTS.md` as the way to produce and gate review surfaces without CI. Hard rule: orchestration only — the scripts call the same CLI commands a user types by hand; no behavior may exist only inside a script, or local and CI surfaces drift.
2. **The picture (M9)** — brainstorm ideas 1 + 2. The schema-validated `change_graph` model (nodes = changed files with churn/lens/status, dashed halo of top unchanged importers from blast-radius facts, edges from the import graph restricted to those nodes, clusters by top-level dir) built from `buildImportGraph()` output — no new parsing. A focused mermaid emitter (`src/diagrams/change-map.ts`, `flowchart LR`, subgraph per cluster, classDef per lens, hard caps with explicit overflow nodes). Embed the map on the comment surfaces reviewers actually see — `renderHumanPrComment` and the sticky summary (`src/render/sticky-summary.ts`), each as a collapsed `<details>` block — plus a new embed in `human_review.md`. (Today's "Change impact" embed lives only on the provider-narrative path: blocked under `--provider mock` and skipped whenever a current `human_review.json` exists — it retires with the hairball.) The spec/requirement-mapping diagram (`pr-change-impact.mmd`) leaves the human surfaces and remains a standalone agent-facing artifact. The guided diff tour: topological reading order over changed files (dependencies first, strongly-connected components collapsed into one read-together leg, alphabetical inside), as a `reading_order` model section rendered after the verdict in `human_review.md` and the cockpit; the sticky comment gets only the first leg. Hard rules: labels escaped via the `diagramLabel`-style sanitizer (unifying the private copy in `pr-change-diagram.ts`) with the body-level fence-close guard kept at every embed point, redaction before render, `scripts/determinism-check.sh` extended with a PR-scope run (today it exercises only repo-scope `all`), no spec/requirement anchors on the map (that is what made v1 unreadable), the tour never includes unchanged files, and the map's left-to-right flow agrees with the tour's numbering.
3. **The cockpit becomes visual (M10)** — brainstorm ideas 3 + 4 + 6. Inline-SVG change map in the HTML cockpit rendered from the same `change_graph` model by a new `src/human/render-svg-map.ts` (hand-rolled layered layout: column = cluster in tour-leg order, stable row sort, fixed `viewBox`, system fonts; node click filters the queue via the existing `data-` attribute pattern; `<title>` hover details so print degrades gracefully). Line-level coverage gutters: grow `CoverageEvidenceHunk` with sorted `uncovered_lines` (capped with an explicit truncated flag), gutter glyph + tint per excerpt line in the cockpit, one summary line with uncovered ranges under markdown excerpts. At-a-glance header strip rendered purely from existing model fields: lens chips that ARE the filter buttons (with counts), the `review_plan` read/skim/defer cut as a stacked CSS bar with minutes, trust counts, and a progress bar fed by the existing checkbox state. Hard rules: **no chart or diagram library and no inlined mermaid.js** in the cockpit; every interpolation through `esc()` (redact-then-escape — module-private in `render-html.ts` today, so export it or lift it to a shared module first); deleted lines never get coverage gutters; color never carries meaning alone (pair with glyph/text, print-safe); byte-deterministic SVG (every coordinate derived from sorted input).
4. **Drift and convergence (M11)** — brainstorm ideas 5 + 7. Architecture-drift facts: parse the *base* content of changed/deleted files (read via `readFileAtRef`, parsed with the import graph's existing extraction), diff resolved import sets, aggregate to module altitude (top-level dir under `src/`), and emit `module_edge_added` / `module_edge_removed` / `import_cycle_created` facts following the `semantic-diff.ts` detector shape — routed into the risk register, an architecture lens, and `kind: new | removed` on `change_graph` edges so both map renderers pick them up for free. Respect rename detection (a moved file re-creating its old edges is not drift); unresolvable imports (path aliases) count as unknown, never as removed. Convergence trend: a `rounds[]` ledger carried forward with the prior packet (round, head SHA, new/resolved/regressed counts from the existing compare output, verdict), rendered as a compact table in the sticky comment and cockpit (last ~8 rounds; full ledger in the artifact). The ledger must work from **both** transports: CI artifacts and the local prior-packet directory from Phase 1. Partial history is normal, never an error. Extend the eval harness with a seeded drift fixture (benign change + new cross-module import, asserted to rank in the top N).
5. **Depth and the trust story (P2 tail)** — brainstorm ideas 8 + 9. Dependency-change tree: attribute each new transitive to the direct dependency that pulled it by parsing lockfile dependency edges (one new deterministic fact field — today's `transitive_added` facts are flat and unattributed), then render the chain as an indented tree in the supply-chain lens (mermaid on GitHub surfaces where a chain exists), install-script flags marked; lockfiles whose edges can't be resolved fall back to the honest flat grouping. Eval scoreboard surfacing: a generated, marker-delimited README block (idempotent regeneration, like the sticky comment's marker upsert — never hand-edit inside the markers) citing cases passed / total per fact class from `eval_scoreboard.json`, plus one cockpit footer line citing the score; the local gate asserts the README block is current.

## Per-phase loop

For each phase, in order:

1. Start from fresh `origin/main` on a new branch named `visual-phase-<n>-<slug>`. Never commit directly to `main`.
2. Implement the phase's ACIDs with tests that reference those ACIDs. **Every new requirement gets at least one test named with its exact ACID** — a requirement whose only evidence is implementation code evaluates as missing on later empty-diff runs and reddens main (the PR #59 lesson, repeated once already).
3. Run the full local gate: from Phase 1 on, `pnpm run local-gate`; for Phases 0–1 themselves, run the underlying commands (`pnpm run lint`, `pnpm run typecheck`, `pnpm run test`, `pnpm run determinism-check`, and the strict empty-diff self-dogfood). All must pass.
4. Dogfood locally — never via CI: from Phase 1 on, `pnpm run local-review` with the mock provider, then read the resulting `human_review.md` **and open `human_review.html` in a browser** as a human reviewer; for phases that ship mermaid, view the rendered diagram (editor preview or a GitHub gist paste). If the phase did not visibly improve the rendered surface, record why in a dogfood feedback file before proceeding.
5. Remove the phase's ACIDs from `quality_gate.allow_missing` and verify the strict gate passes, including the empty-diff run (`all --base HEAD --head HEAD --strict`).
6. **Simplify pass:** before opening or finalizing the PR, run the simplification skill on the change — the skill is named `simplify` if you are a Claude agent, `simplify triad` if you are a Codex agent. Apply its output; re-run the gate after.
7. Open the PR with a body that lists the phase number, the ACIDs covered, the validation commands run, and the dogfood verdict.
8. **Autoland:** run the autoland flow. Respect the repository gates from `.agents/skills/composed-review-loop/SKILL.md` — do not merge unless the gates are satisfied. While GitHub Actions remains billing-blocked, the gate is: clean substantive review verdict **plus** `pnpm run local-gate` green on the current head. Inspect current-head review comments before autolanding (FB-2026-06-08-006). Address review findings and re-run the gates rather than overriding them.
9. **Do not begin the next phase until the current phase's PR is merged into `origin/main`.** After merge, sync `main` and start the next phase from it.

If a phase proves too large for one reviewable PR, you may split it into stacked PRs, but every PR must pass the same simplify + gate discipline, and the phase counts as done only when all its PRs are merged.

## Success condition

All six phases (0–5) merged to `main`; `quality_gate` back to `max_missing: 0, allow_missing: []`. Dogfooding this repo through `pnpm run local-review` — with zero GitHub Actions involvement — shows: the change-impact map (not the requirements hairball) on the PR comment surface and in `human_review.md`, the guided tour after the verdict, the cockpit with the clickable SVG map, line-level coverage gutters, and the at-a-glance strip, drift facts and the rounds trend rendering where their inputs exist, the seeded drift fixture in the eval scoreboard, and a current scoreboard block in the README.
